Behind the Scenes of MyMUN

Hi, I'm Matz, a full-stack developer and the server admin at MyMUN.

My team and I believe in open-source, sharing information and transparency and so it happens that we thought to give you some insight into architectural decisions, the workflow and security measures we use to ensure a stable and continuous functioning of our service. So without further ado, let's start with our first behind-the-scenes tour of the MyMUN project.

Open-Source

Free and especially open-source software was one of the key ingredients that enabled us to build a system with the level of complexity required for a conference management system and social network that MyMUN is. We greatly benefitted from the possibility to customize projects to our demands by forking or working upstream. Besides many little libraries our main software-stack for the web-application itself consists of:

  • Flask

    A very lightweight but extendable micro-framework that served us very well as the backend core.

  • AngularJS

    AngularJS makes the creation of single-page web-applications quite easy, allthough it's "just another JS framework". Well, it actually kind of works.

  • Postgresql

    Just the best.

  • Bootstrap (highly customized of course)

    You might have heard about it.

Architecture

To create a complete system we had to connect these and many other components to make them work seamlessly together . In a very simplified way the main components are a Flask server exposing a REST API und an angularjs client that uses AJAX heavily. Up to this point our whole system seems pretty straightforward but there is more to it than meets the eye. From a developers' perspective there are mainly two things besides coding the application itself: Continuous deployment and development, Data-Security

Continous deployment and development:

The obvious first part of this is a version control system git. In our system the production and development branches are directly connected to a CI service that automatically runs tests, deployes a cloned environment to make sure absolutely everything works under production conditions and finally deployes the live system. The most interesting part is the deployment process, which we use ansible for. Ansible is a lightweight yet very powerful tool every sysadmin and even developer can benefit from. If you haven't heard about it yet, it's basically a linux automation tool for nearly everything ranging from setting up a project on a remote mashine to deploying a big server cluster with many individual components. This leads us to the next topic:

Data-Security:

As an administrator having the responsibility for user's data to stay secure as well as redundant, one has to be prepared for the worst case scenario possible. Let's pretend for a moment the following happens:

A plane crashes into the data-center the MyMUN system runs in and it destroys both the RAID 1 array in the server itself and the separate backup-storage. Fortunately we've got several off-site backups so the data is not lost. But the application is obviously down. What a shame. Then our ansible-powered setup and deployment system comes into play. After uploading the backup, a single shell command would set up a new server and deploy our whole infrastructure within minutes. That's the power of ansible. So even in this kind of scenario we are prepared to get your conferences back to you in the shortest time possible.
